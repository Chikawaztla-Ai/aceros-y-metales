01 — Modelo de Amenazas

Vectores de ataque y mitigación

Clasificados por severidad. Cada amenaza tiene su contramedida implementada en código.

CRÍTICOInyección SQL

Vector: Un atacante envía SQL malicioso a través de los filtros del catálogo, barra de búsqueda o formularios.

Mitigación: Supabase usa queries parametrizados por defecto — supabase.from('products').select().eq('grade', input) nunca concatena strings. Zod valida todos los inputs antes de tocar la DB. Las Edge Functions usan el cliente de Supabase, nunca SQL directo.

CRÍTICOBypass de pagos / manipulación de precios

Vector: Un atacante modifica el precio en el frontend (DevTools) antes de enviar el pedido, o fabrica un webhook de MP con un monto falso.

Mitigación:

1. El precio NUNCA viene del frontend. /api/pagos/crear consulta el precio en Supabase y calcula el total server-side.

2. El webhook de MP valida la firma HMAC con x-signature + el secret de la app. Si no coincide, se rechaza con 401.

3. Después de recibir el webhook, se vuelve a consultar la API de MP para confirmar el monto pagado vs. el total de la orden.

CRÍTICOAcceso no autorizado a datos de otros clientes

Vector: Un cliente autenticado intenta ver pedidos o cotizaciones de otro cliente cambiando el ID en la URL.

Mitigación: RLS (Row Level Security) en Supabase. Cada tabla tiene policies que verifican customer_id = auth.uid(). Aunque alguien manipule la URL o la API, Supabase no devuelve datos ajenos. Esto se verifica a nivel de base de datos, no de aplicación.

ALTOXSS (Cross-Site Scripting)

Vector: Un atacante inyecta JavaScript malicioso en campos como "notas" de cotización, nombre de empresa o blog.

Mitigación: React escapa HTML por defecto (JSX). Nunca usar dangerouslySetInnerHTML con datos de usuario. Sanitizar con DOMPurify los campos de texto libre antes de guardar. CSP header bloquea scripts inline no autorizados.

ALTOCSRF (Cross-Site Request Forgery)

Vector: Un sitio malicioso hace que el navegador del admin ejecute acciones en el dashboard sin su conocimiento.

Mitigación: Supabase Auth usa cookies HttpOnly + SameSite=Lax. Las API Routes verifican el token de sesión en cada request mutante (POST/PUT/DELETE). Cloudflare añade protección adicional.

ALTOBrute force en login

Vector: Un atacante intenta miles de combinaciones de contraseña contra la página de login.

Mitigación: Supabase Auth tiene rate limiting nativo (100 requests/15 min por IP). Cloudflare Bot Management añade otra capa. Después de 5 intentos fallidos, se requiere esperar 60 segundos. Contraseñas mínimo 8 caracteres.

ALTOAbuso de la API de búsqueda IA

Vector: Un atacante o bot hace miles de requests a /api/buscar, generando costos altos en OpenAI.

Mitigación: Rate limit de 20 req/min por IP en esta ruta. Si se excede, fallback silencioso a búsqueda SQL (ILIKE). Monitorear costos diarios de OpenAI con alerta si excede $5 USD/día.

MEDIOSpam en cotizaciones

Vector: Bots envían cientos de cotizaciones falsas, saturando el email del admin.

Mitigación: Rate limit de 5 cotizaciones por IP por hora. Honeypot field invisible en el formulario (si se llena, es bot). Cloudflare Turnstile (CAPTCHA invisible) como última línea.

MEDIOUpload de archivos maliciosos

Vector: Un usuario sube un archivo .exe disfrazado de PDF en el formulario de cotización.

Mitigación: Validar MIME type server-side (solo application/pdf y extensiones DWG/DXF). Limitar tamaño a 10MB. Supabase Storage no ejecuta archivos — solo los almacena. Los archivos se sirven con Content-Disposition: attachment.

MEDIODDoS

Vector: Flood de requests para tumbar el sitio.

Mitigación: Cloudflare está al frente — absorbe DDoS antes de llegar a Vercel. Vercel Edge tiene auto-scaling. Supabase tiene connection pooling con PgBouncer. No hay servidor que tumbar — todo es serverless.

03 — Manejo de Secrets

Variables de entorno y secretos

NUNCA exponer al cliente

SUPABASE_SERVICE_ROLE_KEY — acceso total sin RLS. Solo en API Routes server-side. MP_ACCESS_TOKEN — token de Mercado Pago. OPENAI_API_KEY — costos ilimitados. RESEND_API_KEY — envío de emails.

OK exponer al cliente (NEXT_PUBLIC_)

NEXT_PUBLIC_SUPABASE_URL y ANON_KEY — seguros porque RLS controla el acceso. NEXT_PUBLIC_GA4_ID, META_PIXEL_ID, CLARITY_ID — analytics. NEXT_PUBLIC_WA_NUMBER — WhatsApp.

04 — Cloudflare Config

Configuración de seguridad en Cloudflare

SSL/TLS

Modo: Full (Strict). Vercel tiene su propio certificado, Cloudflare valida contra él. HTTPS forzado.

Firewall Rules

Bloquear requests de países sin clientes (activar si hay abuso). Rate limit: 100 req/10s por IP global.

Bot Management

Bot Fight Mode: ON. Challenge requests sospechosos con Turnstile. Permitir bots buenos (Googlebot, Bingbot).

Caching

Cache de assets estáticos (imágenes, CSS, JS). Bypass cache en API routes y páginas SSR. Purge automático en deploy.

Otorgar solo lo necesario vía policies (ya definidas en Backend Schema) -- Principio: DENY ALL por defecto, ALLOW solo lo explícito -- 4. Desactivar signup público si no se necesita -- Supabase Dashboard → Auth → Settings → Disable Sign Up -- (Solo si el registro será controlado por el admin)

Storage policies

Buckets privados por defecto. Solo product-images y product-docs son públicos (lectura). Uploads requieren auth + MIME validation.

Connection pooling

PgBouncer activado en Supabase. Pool mode: Transaction. Límite de conexiones: 50 (suficiente para MVP). Monitorear en dashboard.

07 — Monitoreo de Seguridad

Qué vigilar en producción

Alertas inmediatas

Webhook de MP rechazado (firma inválida). Rate limit excedido en /api/buscar. Error 500 en /api/pagos. Login fallido > 10 veces en 5 min desde misma IP.

Revisión semanal

Logs de Vercel: buscar 401/403/500 inusuales. Supabase: queries lentas (> 500ms). Cloudflare: IPs bloqueadas y ataques mitigados. OpenAI: costo diario vs. promedio.

08 — Checklist Pre-Launch

Verificar antes de ir a producción

RLS activado en todas las tablas de Supabase. Probar con anon key que NO devuelve datos sin autenticación.
Webhook de MP valida firma — probar con firma falsa, debe devolver 401.
Precios calculados server-side — cambiar precio en DevTools y verificar que el total en el backend no cambia.
HTTPS forzado — http:// redirige a https://. HSTS header presente.
Cloudflare SSL en Full (Strict) — verificar en dashboard de Cloudflare.
.env.local no está en Git — verificar con git log --all --full-history -- .env*
SERVICE_ROLE_KEY no se usa en Client Components — buscar en código fuente.
Rate limits configurados en /api/buscar (20/min), /api/cotizaciones (5/min), /login (10/5min).
Validación Zod en todas las API routes POST/PUT — probar con payloads vacíos, XSS y SQL injection.
Upload files — intentar subir .exe, verificar que se rechaza. Verificar límite de 10MB.
Security headers — verificar con securityheaders.com → debe dar A o A+.
Cloudflare Bot Fight Mode — activado en dashboard.
Passwords — mínimo 8 caracteres configurado en Supabase Auth.
Middleware de auth — intentar acceder a /admin sin sesión → debe redirigir a /login.
RLS cross-check — loguearse como cliente A, intentar acceder a pedido de cliente B vía API → debe devolver vacío.
